# Add full-change implementation mode to OpenSpecImplement

## Why

`:OpenSpecImplement` currently always scopes the agent to a single selected
task, effectively falling back to the next todo task whenever the invocation
does not preserve a nil task selection. This is useful for step-by-step review,
but when a user wants to hand off an entire change for unattended
implementation, they must invoke the command once per task.

The context file generated by `context.lua` always includes "Implement only the
selected task" in its Allowed Scope and "Stop if implementation requires
expanding the selected task scope" in its Stop Conditions. Even when no task is
selected, the stop condition causes the agent to stop as soon as it would need
to move to the next task.

Users should be able to choose between single-task mode (cursor on a task line
in `tasks.md`) and whole-change mode (no task selected), without any
additional command arguments or configuration.

## What Changes

- When a task is selected (cursor on a task line in `tasks.md`), the generated
  context scopes the agent to that task only — existing behavior is unchanged.
- When no task is selected, the generated context instructs the agent to
  implement all tasks in the change in order, without stopping between tasks.
- `OpenSpecImplement` preserves the nil-task mode through its launch path
  instead of silently substituting `parsed.next_task`.
- The goal objective text in `implement.lua` is updated to match the same
  distinction so that `/goal` handoff is also correct in both modes.
- No new command arguments, config keys, or public API changes are required.

## Impact

- Users can trigger whole-change implementation by invoking
  `:OpenSpecImplement` when the current cursor does not resolve to a task line
  in `tasks.md`.
- Single-task mode remains available by positioning the cursor on a task line
  in `tasks.md`.
- The change is confined to task-selection plumbing plus context/prompt
  generation; no behavior changes to layout, model selection, or verification
  commands are required.

## Non-Goals

- Do not add a new command argument or config key for mode selection.
- Do not change the summary buffer UI or task selection behavior.
- Do not change verification commands or launch flow.
- Do not implement task ordering or dependency tracking inside the plugin.
